Grundlagen Git und GitLab

Grundlagen Git, GitLab und Git Tools

  • Git: Versionskontrollsystem (kann per Terminal oder per Software benutzt werden)
  • GitLab: Server für Git (wie GitHub), Zusatzfunktionen wie Issues, Wikis
  • GitHub Desktop: Software, die die Nutzung von Git vereinfacht und visualisiert

Ressourcen zur Nutzung von Git

  • Pro Git book digital kostenfrei verfügbar
  • GitHub Git Cheat Sheet

Git ist ein "distribuiertes Versionskontrollsystem"

  • eine Version unserer Transkription liegt auf GitLab
  • wenn die Transkription bearbeitet werden soll, wird eine lokale Kopie (auf dem Computer) angelegt ("git clone")
  • wenn die Transkription fertig bearbeitet wurde, wird sie gespeichert ("git commit") und diese Version nach GitLab zurückgespielt ("git push")

Git ist ein "distribuiertes Versionskontrollsystem"

  • am nächsten Tag soll die Transkription wieder bearbeitet werden, zwischendurch hat eine Kollegin die Datei bearbeitet und wieder nach GitLab zurückgespielt, jetzt müssen die lokalen Dateien auf dem Computer aktualisiert werden ("git pull")
  • danach können wieder die vorherigen Schritte zum Bearbeiten, Speichern und Zurückspielen durchgeführt werden

Arbeiten mit existierenden Projekten

  • ein Projekt muss einmalig auf den Computern, auf denen damit gearbeitet werden soll, angelegt werden
  • Ein GitLab Projekt lokal nutzen:
  • per Terminal:
    • git clone https://git.uni-paderborn.de/aferger/elan-git-example
  • mit GitHub Desktop
    • File>Clone repository…>URL
    • URL zu GitLab Projekt (URL zu Repository und lokalen Pfad angeben) und gewünschten lokalen Pfad angeben

Erste Git-Konfiguration

  • per Terminal:
    • git config --global user.name "Your Name Comes Here"
    • git config --global user.email you@yourdomain.example.com
  • mit GitHub Desktop
    • File>Options…>Git
    • Name und Email angeben/ändern (gleiche Email wie auch bei der GitLab Anmeldung nutzen)

Änderungen an Dateien (z.B. Transkriptionen) durchführen

  • Sicherstellen dass die lokalen Dateien die neueste Version haben:
    • git pull
    • mit GitHub Desktop
      • Button "Fetch origin"
      • Falls es Änderungen auf GitLab gab wird Button zu "Pull origin"
      • Diesen Button auch drücken, dann werden die Dateien im Ordner auf dem Computer aktualisiert

danach kann wie üblich z.B. mit ELAN an den Dateien gearbeitet werden

Änderungen an Dateien (z.B. Transkriptionen) durchführen

  • Nach Fertigstellen der Arbeiten die Änderungen speichern und nach GitLab zurückspielen
    • git add Filename
    • git commit -m "changed annotation"
    • git push
    • mit GitHub Desktop
      • Änderungen im Ordner werden automatisch erkannt
      • Feld für Message ausfüllen
      • Button "Commit to main" klicken
      • Button "Push origin" klicken

Git History

  • Alle Versionen aller Dateien sind sichtbar, inklusive der Änderungen die vorgenommen wurden

GitLab Projekt

  • dieses Projekt kann wie oben beschrieben als existierendes Projekt genutzt werden ("git clone")
  • existierende Transkriptionen oder andere Dateien können lokal in den durch git Ordner eingefügt werden und (analog zu Änderungen mit Daten) mit GitHubDesktop wie oben beschrieben zu GitLab hinzugefügt werden

Absprache bei der Arbeit an Dateien

  • auch wenn es bei Git grundsätzlich möglich ist, gleichzeitig an Dateien zu arbeiten ist es bei komplexen Dateien wie Transkriptionen häufig trotzdem sinnvoll die Arbeit zeitlich aufzuteilen
  • Merge Konflikte (dazu weiter unten mehr) können so verhindert werden
  • die Arbeit kann in GitLab durch Issues für einzelne Dateien oder auch Tabellen im Wiki organisiert werden

GitLab Issues

  • es können Issues für bestimmte Aufgaben, zum Beispiel Annotationen angelegt werden Beispiel
  • durch "refs #1" in der Nachricht zum commit können diese Issues mit den entsprechenden Änderungen verlinkt werden

Was tun bei Merge-Konflikten?

  • Keine Panik!
  • Mergekonflikte können entstehen wenn gleichzeitig an der gleichen Stelle einer Datei gearbeitet wurde
  • Bei Merge Konflikten entsteht eine Konfliktdatei, wenn es eine ELAN-Datei ist kann sie vorerst nicht mit ELAN geöffnet werden
  • Vorgehen: Konfliktdatei in einem Texteditor öffnen Konfliktmarker entfernen und nur gewünschte Version behalten

Alternative für grafische Darstellung der Konflikte: Meld (nur für Windows), KDiff3

Branches für besondere Änderungen

  • Änderungen separat und können später zusammengefügt werden
  • "new branch"
